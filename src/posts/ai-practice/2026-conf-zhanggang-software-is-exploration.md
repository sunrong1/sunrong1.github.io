# 软件开发是发现和探索：英慕科技张刚在 2026 开发者大会的 6 个核心观点

> 参加了 2026 年开发者大会，听了英慕科技张刚的 talk。这是我听过最反直觉、最"打穿天花板"的开发者认知升级演讲。**它让我重新理解了软件开发的本质。**

---

## 📌 演讲核心一句话

> **"软件开发的核心是发现和探索，不是构建。"**

这一句话，颠覆了我对"开发者"角色的认知。

下面是我整理的 6 个核心观点。

---

## 💎 观点 1：软件开发的本质是"发现 + 探索"，不是"建造"

> **"软件开发是发现和探索"** ⭐⭐⭐⭐⭐

**传统认知：**
- 软件开发 = 把需求翻译成代码
- 程序员是"建筑工人"
- 工作 = 写代码 + 改 Bug

**张刚的视角：**
- 软件开发 = 探索未知问题 + 发现最佳路径
- 程序员是"探险家 + 科学家"
- 工作 = 发现需求 + 探索方案

**我的连接：**

我之前写 PVZ 的时候，**用的正是"探索式开发"**——
- 先用 Vibe Coding 跑通 MVP（探索）
- 跑通后再用 TDD 巩固（确认）
- 提交 PR 时再考虑扩展（再探索）

**28 小时完成的不是"建造"，是"探索"。**

---

## 💎 观点 2：软件开发的三个基本问题，难度完全不同

> **"问题定义、方案设计、软件构建。问题定义难度最高。"**

| 阶段 | 难度 | 角色 | 关键 |
|------|------|------|------|
| **问题定义** | ⭐⭐⭐⭐⭐ | 产品经理 | 把"用户要什么"想清楚 |
| **方案设计** | ⭐⭐⭐ | 架构师 | 选技术、定架构 |
| **软件构建** | ⭐ | 工程师 | 写代码 |

**张刚的论断："产品经理是最重要的岗位"**

**为什么？**

因为**问题定义错，下游全错**：
- 方案设计 = 解决"假问题" → 浪费时间
- 软件构建 = 实现"假需求" → 写完不用
- 唯一不出错的方式：**先把问题定义清楚**

**我的连接：**

在华为推动 AI Agent 项目时，部门专家推 OpenCode，我推 AgentScope。
**最大的冲突不是"哪个工具好"，是"问题是什么"。**

- 部门专家视角：用什么工具？（**方案设计**）
- 我的视角：测试全流程的核心子场景是什么？（**问题定义**）

**这就是我比他们"高一维度"的原因——我站在问题定义层。**

---

## 💎 观点 3：业务熟悉度是"看起来容易、实际难"的关键

> **"看起来比较容易，实际自己对业务比较熟悉，自己已经做过三次。所以关键是：问题是不是已经被良好定义。"**

张刚讲他做 ralph flow 的故事：
- 看着容易（看起来是工具）
- 实际难（需要对业务熟悉）
- 做过 3 次才内化
- **真正的关键是：问题定义清楚了吗？**

**我自己的故事：**

Vibe Coding 28 小时完成 PVZ，看起来是 Claude Code 强，**但我之前 11 年游戏领域知识是隐藏前提**。**问题定义清楚 = 业务熟悉**。

如果让我去 Vibe Coding 一个我不熟悉的领域（比如：航空发动机控制），**28 小时可能搞不定**。

**这就是"业务熟悉"的真正分量。**

---

## 💎 观点 4：人的认知是有限的，LLM 加速认知

> **"软件开发的核心是探索和发现：人的认知是有限的，LLM 加速你的认知。"**

| 维度 | 没有 LLM | 有 LLM |
|------|---------|--------|
| 探索速度 | 慢（受限于个人认知）| 快（LLM 补足未知）|
| 知识边界 | 个人经验 | 集体智慧 |
| 试错成本 | 高 | 低 |

**LLM 不是"更快的写代码工具"，是"更快的认知工具"。**

**我的连接：**

我学 AgentScope V2 的方法：
- 之前：读 API 文档 → 试错 → 慢
- 现在：Claude Code 读源码 → 生成解释 → 加速 10x

**LLM 在加速我的"探索"过程。**

## 💎 观点 5：用 AI 辅助构建认知 — 行动建议：grill-me

> **"可以用 AI 辅助自己构建认知，推荐 grill-me 这个 skill。"**

**grill-me 是什么？**

不是"帮我写"，而是"**把我架在火上烤**"——
- 不断追问，挖到根
- 每个问题给推荐答案
- **逼你暴露认知盲区**

**我已经安装了这个 skill**（路径：`/root/.openclaw/workspace/skills/nox-grill-me/`）。

**我打算立刻用的 3 个场景：**

1. **个人能力沉淀**："请用 grill-me 模式深挖 AI Agent 领域知识"
2. **AgentScope V2 学习**："我已经读完 pytest，请 grill-me 我对 ReAct Agent 的理解"
3. **博客写作**："我想写'AI 时代开发者核心能力'，请 grill-me 挑战我"

**这是"探索式开发"的工具化形态。**

## 💎 观点 6：高质量 = 高生产率（反直觉）

> **"高质量=高生产率"** ⭐⭐⭐⭐⭐

| 传统认知 | 张刚视角 |
|---------|---------|
| 慢工出细活 | 高质量 = 高生产率 |
| 质量 = 加班 | 质量 = 一次做对 |
| 测试浪费时间 | 测试 = 不返工 |

**这一句最适合解释：**

> "如果你写完代码需要大量调试和返工，那说明**质量是低的**。高质量 = 一次做对 = 速度快。"

**我的连接：**

我 LCR 087 复原 IP 用 15 分钟一次过，**没有返工**。这就是"高质量 = 高生产率"。

而我之前某些题做 40 分钟 + 调 Bug 20 分钟 = **60 分钟 + 返工** = 质量低的"假快"。

**写测试、定义清楚、做对，比"快"更值钱。**

## 🎯 6 个观点的"一句话总结"

| # | 观点 | 一句话 |
|---|------|--------|
| 1 | 软件开发 = 发现 + 探索 | 程序员是探险家，不是建筑工 |
| 2 | 三段式 + 问题定义最难 | 产品经理是核心 |
| 3 | 业务熟悉度 = 隐形前提 | 不熟的业务，工具也救不了你 |
| 4 | LLM 加速认知 | AI = 探索加速器 |
| 5 | grill-me skill | 用 AI 烤自己 |
| 6 | 高质量 = 高生产率 | 一次做对 > 反复返工 |
