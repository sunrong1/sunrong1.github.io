      OpenClaw 本地部署实践：模型选择的坑与经验

      Mr.Sun2026年3月26日...大约 5 分钟

      OpenClaw 本地部署实践：模型选择的坑与经验

      摘要： 本文记录了在本地部署 OpenClaw 过程中遇到的模型选择问题，总结了不同规模模型在工具调用能力上的差异，并给出了实用的配置建议。

      关键词： OpenClaw、本地模型、工具调用、AI Agent、大模型部署

      一、背景

      最近在学习和实践 OpenClaw 本地部署，这是一个开源的 AI Agent 框架，支持本地模型调用、工具集成、会话管理等功能。在部署过程中，我遇到了一个关键问题：模型选择。

      最初我尝试使用较小的模型（<14B）进行工具调用，发现效果不理想。经过一番调研和实践，总结了一些经验，希望对同样在探索本地 Agent 部署的同学有所帮助。

      二、OpenClaw 简介

      OpenClaw 是一个开源的 AI Agent 运行框架，核心特性包括：

      • 本地模型支持 - 可对接 llama.cpp、vllm 等本地推理后端
      • 工具调用能力 - 支持 MCP（Model Context Protocol）协议，可调用各种外部工具
      • 会话管理 - 支持多会话、子代理、定时任务等
      • 扩展性强 - 可通过 Skill 机制扩展功能

      我的部署目标：搭建一个本地 AI 助手，能够调用企业微信 API、管理待办事项、查询日程等。

      三、踩坑记录

      3.1 第一次尝试：小模型翻车

      最初为了节省资源，我尝试使用 7B 级别的模型：

      # 配置示例
model: qwen-7b-chat
context_length: 8192

      问题表现：

      • 工具调用格式经常出错
      • 无法正确解析工具定义
      • 返回结果不稳定，有时完全忽略工具

      3.2 第二次尝试：中等模型

      升级到 14B-32B 区间后，情况有所改善：

      model: qwen-32b-chat

      表现：

      • 基本工具调用可以工作
      • 复杂场景下仍有问题
      • 偶尔会忽略工具参数

      3.3 最终方案：70B+ 模型

      使用 70B 以上模型后，效果显著提升：

      model: qwen-72b-chat

      表现：

      • 工具调用稳定可靠
      • 能正确理解复杂工具定义
      • 参数提取准确

      四、核心发现

      经过实践，我总结了以下几点关键发现：

      4.1 工具调用需要专门训练

      工具调用不是模型的天然能力，需要专门训练。

      不是所有模型都支持工具调用（Function Calling）。这个能力需要：

      • 训练数据中包含 API 调用示例
      • 学习工具定义的结构化格式
      • 理解何时调用工具、如何填充参数

      4.2 模型规模与工具调用能力

      模型规模工具调用能力推荐场景
      <14B❌ 较弱不推荐用于 OpenClaw
      14B-32B⚠️ 一般学习/测试可用
      ≥32B✅ 可用工具调用场景
      ≥70B✅✅ 推荐生产环境

      4.3 模型年龄也很关键

      2023 年前的老模型基本不支持工具调用，因为：

      • 训练时 Function Calling 概念还未普及
      • 不理解结构化输出格式
      • 容易忽略工具定义

      4.4 上下文占用问题

      OpenClaw 的工具定义会占用大量上下文：

      每个工具定义 ≈ 500-1000 tokens
10 个工具 ≈ 5000-10000 tokens

      这意味着：

      • 小模型的上下文很快被占满
      • 大模型有更多空间处理工具定义
      • 需要预留足够上下文给对话历史

      五、推荐配置方案

      5.1 开发/测试环境

      model: qwen-32b-chat
context_length: 16384
backend: vllm
gpu: 1x RTX 4090 (24GB)

      适用场景：

      • 本地开发调试
      • 功能验证
      • 学习探索

      5.2 生产环境

      model: qwen-72b-chat
context_length: 32768
backend: vllm
gpu: 2x RTX 4090 (48GB) 或 A100 80GB

      适用场景：

      • 日常使用
      • 多工具调用
      • 复杂任务处理

      5.3 云端备选方案

      如果本地资源有限，可以考虑云端 API：

      model: qwen-max  # 阿里云
# 或
model: glm-4-plus  # 智谱 AI

      优势：

      • 无需本地 GPU
      • 模型能力最强
      • 按需付费

      劣势：

      • 依赖网络
      • 数据隐私考虑
      • 长期使用成本较高

      六、部署实践

      6.1 远程访问配置

      OpenClaw 远程访问需要解决 HTTPS 证书问题，我的方案：

      # Nginx 配置示例
server {
    listen 443 ssl;
    server_name your-domain.com;
    
    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;
    
    location / {
        proxy_pass http://localhost:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

      6.2 Device 连接

      OpenClaw 中的 Device 概念是设备管理的核心：

      # 查看 Device 状态
openclaw device list

# 连接 Device
openclaw device connect <device-id>

      理解 Device 概念后，可以更方便地管理多设备接入。

      七、总结与建议

      7.1 核心结论

      1. 工具调用需要大模型 - ≥32B 可用，≥70B 推荐
      2. 模型年龄很重要 - 优先选择 2024 年后的新模型
      3. 上下文要充足 - 预留足够空间给工具定义
      4. 本地 vs 云端 - 根据资源和隐私需求选择

      7.2 给新手的建议

      1. 不要一开始就追求最小资源 - 工具调用能力比资源节省更重要
      2. 先云端验证，再本地部署 - 用云端 API 验证功能，再考虑本地化
      3. 关注模型更新 - 工具调用能力在快速演进
      4. 记录踩坑经验 - 每次尝试都是宝贵经验

      7.3 后续计划

      • 继续测试不同模型的工具调用表现
      • 探索模型量化方案（在保持能力的前提下降低资源需求）
      • 整理 OpenClaw 最佳实践文档
