AI Agent 面试题目：从理论到工程落地

整理自 2026-05-10 模拟面试练习

一、核心概念理解（10 道高频题）

Q1: 推理与行动的融合（ReAct）

核心思想： 让 LLM 交替进行"推理"和"行动"，通过外部反馈迭代优化。

为什么需要行动：

  1. 信息获取 — 推理需要外部数据，不能凭空猜
  2. 验证假设 — 行动检验推理结论是否正确
  3. 改变状态 — 推理本身不能修改外部状态

局限性：

  • 当 tool 超时/网络错误/返回异常数据时，推理-行动循环断裂
  • 本质上假设外部环境稳定可预测
  • 无法处理完全未见过的错误类型

面试加分回答： 一个 robust 的 Agent 系统必须具备：重试机制、降级策略、异常状态检测。这也解释了从 ReAct 演进到 Voyager 的动力——解决长周期任务中的容错问题。

Q2: Agent 的记忆设计

三层记忆架构：

  • 短期记忆：当前上下文，对话消失前摘要
  • 长期记忆：整体绘画的长期上下文
  • 向量化记忆：存储语义信息，过程记忆

摘要策略优化：

  • 触发时机：上限触发（token 接近上限）+ 自然节点触发（子任务完成/换话题）
  • 摘要风格：结构化模板，第三人称，含任务目标/已完成步骤/当前状态/关键决策点/遗留问题
  • 增量摘要：滑动窗口 + diff，只追加 diff 部分，token 消耗稳定

对比 Voyager 三层：

用户设计Voyager
向量化记忆Episodic Memory
短期记忆 → 摘要 → 长期Working Memory → 写入 Skill Library
长期记忆Skill Library（存可执行技能）

⚠️ 关键遗漏：Skill Library — 存 Agent 学到的可执行技能/方法，不是对话记忆

Q3: 多 Agent 协作（MetaGPT vs AgentVerse）

维度MetaGPTAgentVerse
协作模式角色化流水线（SOP）分层协作（规划-执行-共享）
任务拆解预定义角色+固定流程动态分解，层层递进
结果汇总角色输出拼接分层汇总，逐层整合

Q4: 工具调用失败恢复策略

四层容错设计：

  1. 重试层 — 指数退避重试（网络超时等瞬时错误）
  2. 降级层 — 工具 A 失败，自动切换工具 B（如搜索引擎 A 切搜索引擎 B）
  3. 分解层 — 大工具拆成小工具，定位具体失败节点
  4. 上报层 — 超过重试阈值，记录错误，上报给用户/人工介入

Q5: Agentic RAG

核心改进： 普通 RAG 是"一次检索"，Agentic RAG 是"迭代检索"

维度普通 RAGAgentic RAG
检索次数单次迭代（Agent 决定是否继续查）
判断标准top-k 相似度检索结果是否满足推理需求
适用场景简单问答多跳推理、复杂分析

局限性： 迭代开销大、可能陷入检索死循环

Q6: 长周期任务的状态管理

三层检查点机制：

顶层：任务级 checkpoint（每个子任务完成时存）
中层：步骤级 checkpoint（每 N 个原子操作后存）
底层：工具调用级 checkpoint（关键操作前存）

状态内容： 当前任务目标 / 已完成步骤 / 上下文摘要 / 工具调用历史 / 错误日志

恢复策略： 从最近的 checkpoint 恢复，保留已完成的步骤只重跑失败分支

Q7: 架构演进（ReAct → Voyager）

演进逻辑：

ReAct（单步循环）
  ↓ 任务变复杂
MetaGPT（角色流水线）
  ↓ horizon 变长
AgentVerse（分层协作）
  ↓ 需要长期记忆
Voyager（终身学习三层架构）

核心突破： 从"每次任务从零开始"到"积累技能和经验"

下一代方向： 多 Agent 自主协作 + 自我反思改进

Q8: Computer Use vs 传统 GUI 自动化

维度Computer Use（多模态）Selenium（传统）
核心优势无需 DOM 接口，直接理解屏幕精确控制 DOM
适应性能处理任意 UI 变化依赖 DOM 结构
稳定性瓶颈视觉理解错误 / 细微按钮误点依赖 DOM 稳定性

Q9: 测试用例生成场景的工具设计

工具集设计：

  • 代码解析工具（读被测代码）
  • 测试框架调用工具（执行测试）
  • 断言生成工具（生成验证点）
  • 覆盖率统计工具（衡量质量）

可复现性： 固定 seed + 版本快照 + 结果 Diff 对比

线上评估： 通过率/覆盖率/漏测率/人工验收评分

Q10: Prompt Injection 防御

三种核心防御手段：

  1. 输入 sanitization — 过滤/转义用户输入中的指令性内容
  2. 权限隔离 — Agent 权限最小化，核心操作需二次确认
  3. 任务边界限制 — Agent 只能在限定范围内行动，超出需显式授权

二、记忆系统设计（深化版）

长期记忆写入时机策略

优先级内容类型写入策略
高工具调用失败立即写入
高用户明确偏好立即写入
中关键决策点子任务完成时写入
低日常闲聊session 结束统一摘要再决定

核心原则：错误必记、偏好必记、经验可选记

向量检索召回质量优化

混合检索架构：

Query → Dense（向量相似度）──┐
         ↓                   ├→ Rerank → 最终结果
      Sparse（BM25）───────-┘

配合元数据过滤： 时间范围、任务类型、Agent ID

Rerank 模型： BAAI/bge-reranker-base（百毫秒级延迟）

长期记忆选型

  • 精确索引层： Redis / PostgreSQL，主键 exact match
  • 向量检索层： Qdrant，语义模糊匹配
  • 写入时： 根据内容类型自动分流
  • 查询时： 先精确后向量，精确未命中再走向量兜底

新错误查询策略

新错误走两条路：

  1. 精确查长期记忆 — 看是否见过但表达不同
  2. 外部搜索兜底 — StackOverflow、行业知识库、主动上网搜

同时新错误写回记忆，等效下次类似错误时能检索到。

三、面试核心原则

记忆系统的本质不是存得越多越好，而是：什么时候查、查哪层、怎么过滤。这三件事没想清楚，加再多存储也是噪音。

ReAct 的脆弱性：本质假设外部环境稳定可预测。真实生产需要：重试 + 降级 + 异常检测。

面试加分：不仅理解原理，还要看到局限性，有工程思考。
